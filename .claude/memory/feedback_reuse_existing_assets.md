# 원칙

새 최적화를 제안하기 전에, **이미 프로젝트에 존재하는 집계/캐시 자산** 을 먼저 훑는다:
- Materialized View (kepco_map_summary 등)
- 공통 테이블 (kepco_addr 같은 마스터)
- 기존 인덱스
- 기존 RPC / 함수

## Why

2026-04-22 search_kepco 최적화 사례:
- 사용자 검색이 2초 걸려 최적화 요청
- Claude 는 "기계적 최적화" 만 제안: 함수형 인덱스 추가, scalar subquery 재작성 등
- 시나리오 분석에서 제안 쿼리가 광범위 검색 시 오히려 악화됨 확인
- 사용자가 "시나리오부터 분석해라" 압박 → 그제서야 `kepco_map_summary` MV 가 이미
  리 단위 집계(total, addr_do/si/gu/dong/li, lat, lng) 다 담고 있음을 발견
- MV 직접 조회로 1395ms → 125ms (11배 개선), 모든 시나리오 안정적

고정관념: "MV 는 지도 마커용" 이라는 이름/위치 편견이 재활용 가능성을 가림.

## How to apply

- 성능 문제/쿼리 재작성 시 첫 단계: **"이 정보가 이미 어딘가에 집계되어 있지는 않나?"**
- 테이블/뷰 목록을 먼저 확인 (pg_matviews, pg_views, 마이그레이션 파일)
- 구조적 재활용 > 기계적 최적화 (인덱스 추가, 서브쿼리 재작성, 힌트)
- 인덱스/쿼리 최적화는 자산 재활용이 정말 불가능할 때의 차선
- 사용자가 "시나리오 분석부터" / "확실히 개선되나?" 라고 물으면: 의심 깊게 재검토. 확률 높게 내가 놓친 게 있다.
