새 테이블/데이터셋에 외부 API 를 붙일 때, 코드를 쓰기 **전에** 반드시 아래 세 질문에 답한다:

1. **이 테이블의 PK/기준값은 무엇인가?**
2. **이 외부 API 는 그 PK 로 직접 조회 가능한가?** (문서 확인 필수, 기억 신뢰 금지)
3. **다른 키로 조회해야 한다면, 변환 과정에서 정보 손실 또는 의미 변질은?**

**Why:** 2026-04-23 bjd_master 좌표 채우기 사건. bjd_master 의 PK 는 `bjd_code` 인데, 나는 `fill_kepco_coords.py`(평문 주소 기반 search API 호출) 로직을 관성적으로 복사해서 `sep_1~5` → VWorld search API 로 조회함. bjd_code 를 가지고 있는데 평문으로 돌려 맞추는 꼴. L3 읍/면에서 12% bjd 불일치 발생. bjd_master 를 만든 목적("bjd_code 를 유일한 기준값으로 삼아 평문 오염에서 벗어나기")과 구현(다시 평문으로 조회) 이 정면 모순이었음.

**How to apply:**
- 기존 스크립트를 포팅할 때: "기준값이 같은가?" 먼저 확인. 다르면 **로직 전면 재설계** 필요.
- 외부 API 문서 검색부터 하고 내 기억에 있는 엔드포인트만으로 설계하지 말기.
- "이 설계의 본래 목적" 을 한 줄로 쓰고, 구현 각 단계가 그 목적을 어기지 않는지 되새기기.
