실수/오류를 발견했을 때 해결책을 고를 때, **극단 방안(전부 리셋 / 전면 재작성) 을 제안하기 전에 반드시 중간 방안을 먼저 검토**한다:

- **기존 산출물을 부분 보존** 할 수 있는가? (예: 일부 정확한 데이터는 유지, 부정확한 것만 교체)
- **"완벽"이 아니라 "목적에 맞는 실용"** 수준이면 충분한가?
- **사용자 이전 요청** 과 일관성 있는가? (기존 판단을 180도 뒤집는 건 의심 신호)

**Why:** 2026-04-23 bjd_master 좌표 사건. search API 기반 kepco_addr 좌표가 "정확한 중심점이 아니다" 는 걸 깨닫자 즉시 "전부 리셋 + 전수 data API 재조회(~2.5시간)" 를 제안. 하지만 사용자가 "kepco_addr 기준 이전 + 나머지만 VWorld" 라는 실용적 절충안을 제시함 — 지도 마커 용도엔 충분한 정확도 + VWorld 호출 최소화 + 시간 절약. 나는 중간지점을 아예 못 떠올리고 극단 방안으로 점프함.

**심리 패턴 (매몰비용 역효과)**: 내 실수를 "통째로 지워야 깔끔" 하다고 느끼는 결벽. 실제로는 기존 작업 중 정확/쓸만한 부분을 구분해 살려야 더 낫다.

**How to apply:**
- 해결책 제시 전에 "**부분 살리기 가능한 경계선은 어디인가**" 자문.
- 사용자 이전 결정/검증과 **판단이 뒤집히는 순간** 스스로 경고. 무엇이 바뀌어서 판단이 달라졌는지 명시 (또는 이전 판단이 틀렸던 이유 설명).
- "완벽한 정답" vs "목적 충족" 분리. 지도 마커 / 시각화 / 단순 집계엔 실용이면 충분.
- 재작업 범위를 말할 때 "전부 / 일부 / 혼합" 3안 먼저 비교.
