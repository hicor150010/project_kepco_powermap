description: 빌드/타입체크 통과 ≠ 푸시 승인. 실제 기능 성능 테스트 후 푸시
type: feedback
---

커밋은 바로 가능하지만 **푸시는 성능 테스트(실사용) 통과 후에만**.

**Why:** 빌드/타입체크만 통과해도 실제 브라우저에서 느리거나 동작 안 할 수 있음. 특히 외부 API 연동 기능(VWorld, 카카오 등)은 실측 성능까지 확인해야 배포 가치 있음. 푸시 = Vercel 자동 배포라 롤백 비용도 큼.

**How to apply:**
1. 코드 변경 완료 후: 타입체크 + 빌드 자체 확인
2. 커밋 (local 만) — 이력 확보
3. **성능 테스트**: 브라우저 실행, 실제 기능 사용, 속도/정확성 확인
4. 통과 → push
5. 실패 → 수정 + 다시 테스트 (amend 또는 추가 커밋)

**교훈** (2026-04-21):
1차 1단계 개발 후 빌드 통과만 보고 `git commit && git push` 일괄 실행 → 사용자가 "푸시는 하면안대" 로 원칙 일깨움. 로드뷰/수집기 증설 때도 동일 원칙이었음 (당시엔 사용자가 "커밋푸시하자" 명시 요청이라 OK).

**기본값: "커밋만" 지시가 없어도 푸시는 묻고 진행**.
