## 1. 시스템 전체 구조

### 현재 (AS-IS)
```
[데스크탑 크롤러]                        [웹 서비스]
KEPCO API → 엑셀 다운로드 ──(수동 업로드)──→ Supabase DB → 지도 표시
     (Python/Tkinter)                      (Next.js/Vercel)
```

### 목표 (TO-BE)
```
[GitHub Actions]                         [웹 서비스]
KEPCO API ──→ Supabase DB 직접 INSERT ──→ 지도 표시
  (Python, 무료)          (엑셀 중간 단계 제거)    (Next.js/Vercel)

[데스크탑 크롤러] ← 별도 유지 (의뢰자용 독립 도구)
```

### 핵심 변화
- **엑셀 다운로드 → 수동 업로드** 과정이 **완전히 제거**됨
- GitHub Actions가 KEPCO API를 직접 호출하고 결과를 Supabase DB에 INSERT
- 웹은 기존 그대로 DB에서 읽어서 지도 표시
- 데스크탑 크롤러는 의뢰자가 직접 사용하는 독립 도구로 유지

## 2. 프로젝트 구성

### 2.1 kepco_web_map (웹 서비스) — Phase 1 완료

| 항목 | 내용 |
|---|---|
| 기술 스택 | Next.js 16 + React 19 + TypeScript + Supabase + Tailwind CSS 4 |
| 호스팅 | Vercel (무료) |
| DB | Supabase (무료, 500MB) |
| 지도 | Kakao Maps SDK |
| 지오코딩 | VWorld API (무료, 만료 2026-10-08) |
| 인증 | Supabase Auth (이메일+비번, 회원가입 X) |

#### 라우팅 구조
```
app/
├── page.tsx                ← 메인 지도 (인증 필수)
├── login/page.tsx          ← 로그인
├── admin/
│   ├── upload/page.tsx     ← 엑셀 업로드 (admin만)
│   └── users/page.tsx      ← 계정 관리 (admin만)
└── api/
    ├── upload/route.ts     ← 엑셀 처리 API
    ├── map-summary/        ← Light 조회 (리 단위 집계)
    ├── location/           ← Heavy 조회 (해당 리 raw 데이터)
    ├── search/             ← 주소 검색
    ├── geocode/            ← 지오코딩
    └── admin/users/        ← 계정 CRUD
```

#### DB 스키마 (4개 핵심 테이블)
| 테이블 | 역할 |
|---|---|
| `kepco_data` | 메인 raw 데이터 (주소, 좌표, 시설, 용량, 여유상태) |
| `geocode_cache` | 주소→좌표 영구 캐시 (재호출 방지) |
| `kepco_map_summary` | Materialized View (리 단위 집계, Light 조회용) |
| `user_roles` | 사용자 권한 (admin/viewer) |

#### 완료된 기능
- 인증/권한 (로그인, admin/viewer, 계정 관리)
- 엑셀 업로드 (양식 검증, 지오코딩, upsert, 결과 리포트)
- 지도 (마커, 클러스터, 필터, 검색, 거리재기, 상세 모달, 그룹보기)
- 상세 내용은 [PLAN.md](./PLAN.md) 참조

### 2.2 kepco여유선로용량 (데스크탑 크롤러)

| 항목 | 내용 |
|---|---|
| 기술 스택 | Python 3 + Tkinter + requests |
| 빌드 | PyInstaller (exe 배포) |
| 대상 | 의뢰자가 직접 사용하는 독립 도구 |
| 위치 | `project_crawling/kepco여유선로용량(의뢰자_K1567361460)/src/` |

#### 파일 구조
| 파일 | 역할 |
|---|---|
| `main.py` | Tkinter GUI (주소 선택, 3가지 추출 모드, 로그, 진행률) |
| `api_client.py` | KEPCO API 클라이언트 (세션관리, 재시도, 속도제어, 연속에러감지) |
| `crawler.py` | 크롤링 엔진 (6단계 계층 순환: 시도→시→구→동→리→번지) |
| `checkpoint.py` | 체크포인트 저장/로드 (settings.json) |
| `excel_export.py` | 엑셀 변환 (openpyxl, 그룹별 색상) |

#### KEPCO API 엔드포인트
| API | 경로 | 용도 |
|---|---|---|
| 시/도 목록 | `/ew/cpct/retrieveAddrInit` | 시/도 목록 반환 |
| 주소 계층 | `/ew/cpct/retrieveAddrGbn` | gbn: 0=시, 1=구, 2=동, 3=리, 4=번지 |
| 용량 검색 | `/ew/cpct/retrieveMeshNo` | 주소 기반 배전선로 여유용량 조회 |
| 상세 데이터 | `/ew/cpct/retrieveDl` | STEP 01/02/03 접속예정 건수/용량 |

#### 추출 모드 3가지
| 모드 | 설명 |
|---|---|
| 즉시 추출 | 수동 시작 → 완료 시 자동 저장 |
| 스케줄 추출 | 요일/시간 지정 → 주기적 자동 실행 |
| 이어서 추출 | 체크포인트 로드 → 마지막 위치부터 재개 |

### 2.3 kepco-crawl-runner (GitHub Actions 자동 크롤링) — Phase 2 미착수

| 항목 | 내용 |
|---|---|
| 저장소 | GitHub Public repo (`hicor1/kepco-crawl-test`) |
| 실행 환경 | Ubuntu (GitHub Actions runner) |
| 비용 | **무료** (Public repo = Actions 분 제한 없음) |
| 시간 제한 | job당 최대 6시간 (시/도별 분할로 충분) |
| 트리거 | 수동 (workflow_dispatch) / Cron 스케줄 |

#### API 접근 검증 결과 (2026-04-09)
| 항목 | 결과 |
|---|---|
| 러너 위치 | 미국 버지니아 (Microsoft Azure) |
| KEPCO 사이트 접근 | ✅ HTTP 200 |
| KEPCO API 호출 | ✅ 17개 시/도 정상 반환 |
| **결론** | **해외 IP 차단 없음, GitHub Actions에서 크롤링 가능** |

#### 아키텍처
```
[GitHub Actions Runner]
  ├── api_client.py     ← 데스크탑 크롤러에서 재사용
  ├── crawler.py        ← 데스크탑 크롤러에서 재사용
  ├── crawl_to_db.py    ← 신규: CrawlResult → Supabase INSERT
  └── .github/workflows/
      └── crawl.yml     ← 워크플로우 (스케줄/수동 트리거)
```

#### 동작 흐름
```
트리거 (수동 또는 Cron)
  ↓
Python 환경 셋업
  ↓
api_client.py + crawler.py로 KEPCO API 크롤링
  ↓
crawl_to_db.py: 결과를 Supabase REST API로 UPSERT
  ↓
Materialized View REFRESH
  ↓
웹 지도에서 즉시 반영
```

#### 민감 정보
| 항목 | 저장 위치 | Public repo에서 노출? |
|---|---|---|
| Supabase URL | GitHub Secrets | ❌ |
| Supabase Service Key | GitHub Secrets | ❌ |
| KEPCO API 엔드포인트 | 코드 | 공개 API라 무관 |
